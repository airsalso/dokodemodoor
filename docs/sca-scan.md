# SCA Scan: Google OSV-Scanner vs. Dokodemodoor AI Scanner

본 문서는 Google의 순수 OSV-Scanner(엔진)와 Dokodemodoor에서 구현한 AI 기반 `osv-scanner.mjs`의 성능 및 기능 차이점을 비교 분석합니다.

---

## 🚀 개요

Google의 **OSV-Scanner**는 오픈소스 취약점(SCA) 엔진 중 최상위권의 정확도와 속도를 자랑하는 도구입니다. **Dokodemodoor**는 이 강력한 엔진을 그대로 활용하면서 그 위에 **AI 지능(Agentic Layer)**을 결합하여, 단순한 탐지를 넘어 실제 위협을 판별하는 보안 전문가 수준의 분석을 제공합니다.

---

## 🔍 핵심 차이점 비교분석

### 1. 분석의 목적: "현황 파악" vs "위협 판별"
*   **Google OSV-Scanner (엔진)**: 락파일(`package-lock.json` 등)을 읽어 취약점이 있는 라이브러리 목록을 제공합니다.
    *   *결과*: "A 패키지 1.2.0 버전은 CVE-2023-XXXX에 취약합니다."
*   **Dokodemodoor (AI 스캐너)**: 목록을 받은 후 AI 에이전트가 소스 코드를 직접 분석하여 해당 라이브러리의 취약한 함수가 실제로 호출되고 있는지 확인합니다.
    *   *결과*: "A 패키지가 사용 중이지만, 소스 코드 확인 결과 공격자가 제어할 수 있는 입력값이 해당 취약 함수로 흐르지 않으므로 현재는 안전합니다."

### 2. 호출 그래프(Call Graph) 분석의 범용성
*   **Google OSV-Scanner**: Go, Rust 등 일부 언어에 최적화된 고도의 호출 그래프 분석 기술을 보유하고 있으나, 지원 범위가 특정 생태계에 한정될 수 있습니다.
*   **Dokodemodoor**: LLM의 코드 이해 능력을 활용하여 **언어와 상관없이** 호출 맥락을 분석합니다. AI가 `import` 문부터 시작하여 데이터의 흐름(Taint Analysis)을 논리적으로 추적하므로, 구글 엔진이 공식 지원하지 않는 언어에서도 실제 노출 여부를 판단할 수 있습니다.

### 3. 대응 방안의 구체성
*   **Google OSV-Scanner**: 패키지 업데이트를 권장하며 수정된 최신 버전 정보를 제공합니다.
*   **Dokodemodoor**: 단순히 버전을 올리라는 조언을 넘어, 소스 코드의 맥락에 맞는 **맞춤형 수정 코드**를 제안합니다. 라이브러리 충돌 가능성을 고려하거나 특정 함수 호출 시 필터링을 추가하는 등 실무적인 조치를 가이드합니다.

## 💡 비교 요약표

| 구분 | Google OSV-Scanner | Dokodemodoor AI Scanner |
| :--- | :--- | :--- |
| **정체성** | 데이터 기반의 정밀한 **엔진** | 엔진 + 소스코드 분석 **AI 전문가** |
| **결과물** | 취약점 라이브러리 목록 (JSON/Table) | 실제 위협 시나리오 + 코드 근거 리포트 |
| **분석 범위** | 메타데이터 기반 버전 대조 | **소스 코드 정밀 추적(Taint Analysis)** |
| **사용자 역할** | 보고서를 보고 위험도를 직접 판단해야 함 | AI의 위험도 판별 결과를 검토 및 적용 |
| **최고의 장점** | 속도와 광범위한 취약점 DB | **실제 공격 가능성(Exploitability) 확인** |

---

## ✅ 결론

Google의 OSV-Scanner는 **"최고의 재료(데이터)"**이며, Dokodemodoor의 AI 스캐너는 그 재료를 바탕으로 **"완성된 요리(인사이트)"**를 제공합니다.

단순히 취약점이 포함된 패키지를 나열하는 것을 넘어, **"우리 서비스가 이 취약점 때문에 실제로 공격당할 수 있는가?"**라는 질문에 대한 답을 찾는다면 Dokodemodoor의 AI 기반 통합 점검이 훨씬 강력한 솔루션이 됩니다.
